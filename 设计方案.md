
# CardioCheck 数据集选型与多数据集集成学习原型方案

## 一、业务目标

CardioCheck 的目标不是做单一学术数据集的分类实验，而是做一个可交互的产品原型：

- 用户输入体检指标、化验结果、既往史、生活方式信息
- 系统输出心血管风险概率
- 系统给出风险分层和关键影响因素提示

因此，数据集选择标准应优先满足以下条件：

1. 输入字段尽量来自体检单、检验单或常规问诊
2. 标签能够映射为统一的二分类风险标签或事件风险标签
3. 字段能够被统一到一套产品表单 schema
4. 样本量足以支撑概率输出和基础校准

## 二、适合 CardioCheck 的数据集选型表

| 数据集 | 适配度 | 典型字段是否贴近体检输入 | 样本规模 | 主要预测目标 | 优点 | 主要问题 | 结论 |
|---|---|---:|---:|---|---|---|---|
| Cardiovascular Disease Dataset | 很高 | 很贴近 | 约 7 万 | 心血管疾病风险二分类 | 字段像真实体检表，样本量够，适合训练概率模型 | 多为公开整理版，临床定义和采样背景不如严肃队列研究规范 | 第一版主数据集 |
| Framingham Heart Study | 很高 | 很贴近 | 数千级 | 未来几年冠心病风险 | 医学解释性强，天然适合做风险概率 | 字段版本较多，需要统一清洗和映射 | 第二主数据集，用于增强长期风险建模 |
| UCI Heart Disease Cleveland | 中等 | 部分贴近 | 303 | 当前心脏病风险/诊断 | 经典、干净、容易上手，适合 baseline | 样本太小，部分字段普通用户拿不到 | 保留为 baseline 和对照集 |
| Statlog Heart | 中等 | 部分贴近 | 270 | 心脏病分类 | 和 UCI 类似，适合对照实验 | 样本更小，业务价值有限 | 可选补充，不作为核心数据源 |
| Heart Failure Clinical Records | 中等偏低 | 部分贴近 | 299 | 心衰死亡或预后风险 | 如果以后做心衰子模块有价值 | 任务太窄，不等于泛心血管风险 | 暂不纳入首版原型 |
| MIMIC-III / MIMIC-IV | 中等 | 不完全贴近 | 很大 | 住院死亡、再入院、临床事件风险 | 更接近真实医疗数据，研究价值高 | 清洗复杂，很多字段普通体检用户无法提供 | 适合医院端升级版本 |
| UK Biobank | 高 | 贴近但复杂 | 超大规模 | 长期心血管事件风险 | 高质量、大样本、多模态 | 获取门槛高，工程和合规成本高 | 远期升级路线 |
| PTB-XL | 低 | 不贴近 | 2 万+ ECG | 心电异常/疾病识别 | ECG 场景价值高 | 需要心电图，不是手输体检指标场景 | 当前不适合 |
| MIT-BIH Arrhythmia | 低 | 不贴近 | 小到中等 | 心律失常检测 | 经典 ECG 基准集 | 输入形态不符合你的业务 | 当前不适合 |
| EchoNet-Dynamic | 低 | 不贴近 | 上万视频 | 超声心动图功能评估 | 影像价值高 | 依赖超声视频和专科设备 | 当前不适合 |

## 三、推荐纳入原型的多数据集组合

为了兼顾可落地性、样本规模和医学解释性，CardioCheck 第一版建议采用以下三层数据组合：

| 层级 | 数据集 | 作用 |
|---|---|---|
| 主训练集 | Cardiovascular Disease Dataset | 提供与体检表最接近的主数据分布 |
| 风险增强集 | Framingham Heart Study | 增强长期风险预测能力和医学解释性 |
| 基线对照集 | UCI Heart Disease Cleveland | 提供经典 benchmark，对照模型稳定性 |

推荐结论：

- 第一阶段先接入 Cardiovascular Disease Dataset + Framingham
- UCI Cleveland 继续保留，用于基线比较，不作为最终主模型的唯一来源

## 四、产品统一输入字段 Schema

多数据集集成学习的前提，不是把不同 CSV 直接拼起来，而是先建立一套统一字段层。首版建议统一为以下字段：

| 产品字段 | 类型 | 是否建议首版必填 | 说明 |
|---|---|---|---|
| age | 数值 | 是 | 年龄 |
| sex | 类别 | 是 | 性别 |
| height_cm | 数值 | 否 | 身高 |
| weight_kg | 数值 | 否 | 体重 |
| bmi | 数值 | 否 | 可由身高体重换算 |
| systolic_bp | 数值 | 是 | 收缩压 |
| diastolic_bp | 数值 | 是 | 舒张压 |
| total_cholesterol | 数值 | 否 | 总胆固醇 |
| hdl | 数值 | 否 | 高密度脂蛋白 |
| ldl | 数值 | 否 | 低密度脂蛋白 |
| triglycerides | 数值 | 否 | 甘油三酯 |
| fasting_glucose | 数值 | 否 | 空腹血糖 |
| hba1c | 数值 | 否 | 糖化血红蛋白 |
| smoker | 类别 | 否 | 是否吸烟 |
| alcohol | 类别 | 否 | 是否长期饮酒 |
| active | 类别 | 否 | 是否经常运动 |
| diabetes_history | 类别 | 否 | 糖尿病史 |
| hypertension_history | 类别 | 否 | 高血压史 |
| family_history_cvd | 类别 | 否 | 心血管家族史 |

说明：

- 如果某个数据集没有某些字段，不强行补齐，允许缺失后由预处理层填补
- 如果某个数据集的字段名称不同，先做映射再进入统一 schema
- 如果某些字段无法由普通用户提供，不进入首版主表单

## 五、多数据集集成学习方案

### 方案目标

目标不是简单“拼数据后训练一个模型”，而是做分层集成：

1. 数据集内模型：每个数据集训练一组 base learner
2. 数据集间集成：把不同数据集训练出的概率输出再做二层融合
3. 最终输出：统一风险概率 + 风险等级 + 主要特征提示

### 推荐模型结构

| 层级 | 方案 | 说明 |
|---|---|---|
| 第一层 | RandomForest、GradientBoosting、LogisticRegression | 在单个数据集内学习非线性与线性风险模式 |
| 第二层 | Soft Voting 或 Stacking | 融合不同模型、不同数据集的预测概率 |
| 输出层 | 概率校准 + 风险分层 | 输出低风险、中风险、高风险 |

### 推荐实现方式

方案 A：统一字段后合并训练

- 将 Cardiovascular Disease Dataset、Framingham、UCI Cleveland 映射到统一 schema
- 统一标签为 risk_label，定义为 0 或 1
- 合并后训练一个强基线模型
- 优点是实现简单，适合快速验证
- 缺点是不同数据集分布差异会被直接混合

方案 B：每个数据集先独立建模，再做二层集成

- 为每个数据集分别训练一个 pipeline
- 每个 pipeline 输出一个风险概率
- 将多个概率和少量核心字段一起输入 meta-model
- meta-model 输出最终概率
- 这是更适合产品原型升级的方案

对 CardioCheck 的建议：

- 第一版先做方案 A，跑通端到端流程
- 第二版升级到方案 B，提升跨数据集稳健性

## 六、标签统一策略

不同数据集的标签定义并不完全一致，因此需要统一：

| 数据集 | 原标签 | 建议统一方式 |
|---|---|---|
| Cardiovascular Disease Dataset | cardio 0/1 | 直接使用 |
| Framingham | TenYearCHD 0/1 | 直接使用 |
| UCI Heart Disease Cleveland | 0 到 4 | 转换为 0=低风险，1=有风险 |

统一标签命名建议：

- target
- 或 risk_label

业务定义建议：

- 0 表示当前无明显心血管高风险信号
- 1 表示存在心血管疾病风险或未来事件风险

注意：

- 这个统一标签适合做产品原型
- 如果后续进入临床级应用，需要按疾病定义重新拆分任务，例如冠心病风险、心衰风险、再入院风险分别建模

## 七、原型训练路线图

### 第一阶段：可运行原型

目标：做出可演示的风险预测系统。

建议步骤：

1. 接入 Cardiovascular Disease Dataset
2. 接入 Framingham
3. 建立字段映射表和统一 schema
4. 统一标签后训练 soft voting 模型
5. 输出概率、阈值分类和特征重要性

### 第二阶段：稳健集成

目标：提升多来源数据的鲁棒性。

建议步骤：

1. 各数据集分别训练子模型
2. 做 out-of-fold 概率输出
3. 训练 stacking meta-model
4. 引入概率校准
5. 比较单数据集模型与多数据集集成模型效果

### 第三阶段：产品化

目标：变成可交互产品原型。

建议步骤：

1. 设计统一输入表单
2. 模型服务化，输出 JSON 概率结果
3. 增加风险等级说明
4. 增加字段缺失提示和输入校验
5. 增加结果解释模块

## 八、对当前 CardioCheck 的最终建议

如果 CardioCheck 现在就要做成产品原型，最合适的技术路径是：

1. 主数据集采用 Cardiovascular Disease Dataset
2. 增强数据集采用 Framingham Heart Study
3. 保留 UCI Cleveland 作为 baseline
4. 统一字段到体检表单 schema
5. 先做 soft voting 集成，再升级到 stacking

不建议当前首版纳入的方向：

- ECG 波形类数据
- 超声视频类数据
- ICU 住院过程型数据

原因很简单：这些数据的输入方式不符合“用户自己输入体检和医疗检测指标”的产品形态。

## 九、首版原型最小可行方案

最小可行版本建议如下：

| 模块 | 方案 |
|---|---|
| 输入 | 年龄、性别、血压、胆固醇、血糖、BMI、吸烟、饮酒、运动 |
| 数据集 | Cardiovascular Disease Dataset + Framingham |
| 模型 | RandomForest + GradientBoosting + LogisticRegression 的 soft voting |
| 输出 | 风险概率、风险等级、关键影响因素 |
| 部署形态 | 命令行原型或简单 Web 表单 |

这套方案适合当前阶段，因为它兼顾了：

- 数据可获得性
- 字段可解释性
- 建模复杂度可控
- 向产品原型迁移成本低

## 十、当前已落地的第一步

当前仓库已补充首个可执行脚本：

- src/download_datasets.py

用途：

- 下载 CardioCheck 首版原型所需的数据集
- 默认目标数据集为 Cardiovascular Disease Dataset、Framingham、UCI Cleveland
- 下载结果会保存到 data/raw
- 脚本会输出下载清单 download_manifest.json

安装依赖：

```powershell
pip install -r requirements.txt
```

下载全部首版数据集：

```powershell
python src/download_datasets.py
```

只下载 UCI Cleveland：

```powershell
python src/download_datasets.py --dataset uci-cleveland
```

只下载主数据集和 Framingham：

```powershell
python src/download_datasets.py --dataset cardiovascular-disease --dataset framingham
```

如果 OpenML 自动定位失败，可以传入公开 CSV 镜像地址：

```powershell
python src/download_datasets.py --dataset framingham --framingham-url <你的CSV地址>
python src/download_datasets.py --dataset cardiovascular-disease --cardio-url <你的CSV地址>
```

说明：

- UCI Cleveland 使用官方 UCI 地址直接下载
- Framingham 和 Cardiovascular Disease Dataset 默认优先尝试 OpenML 自动检索
- 如果公开镜像来源后续调整，优先使用命令行参数覆盖数据源

## 十二、第三步已落地：字段映射配置外置

当前已经将字段映射从代码中抽离到独立配置文件：

- config/dataset_mapping.json

prepare_unified_dataset.py 新增参数：

```powershell
python src/prepare_unified_dataset.py --mapping-config config/dataset_mapping.json
```

配置结构说明：

- datasets: 数据集列表
- name: 数据集名称
- input_file: 原始 CSV 文件名（位于 data/raw）
- output_file: 清洗后输出文件名（位于 data/processed）
- mappings: 统一字段映射规则

规则类型：

- type=constant: 常量赋值
- type=column: 从原始列读取并按 transform 变换

当前支持 transform：

- identity
- to_numeric
- binary
- age_days_to_years
- gender_2_to_male_binary

扩展新数据集时，优先流程：

1. 将新数据集 CSV 放入 data/raw
2. 在 config/dataset_mapping.json 新增一个 datasets 条目
3. 配置 input_file、output_file 和 mappings
4. 重新运行 prepare_unified_dataset.py

这样可以做到新增数据集时不改清洗主代码，仅维护配置。

## 十三、第四步已落地：预测接口脚本

当前已新增预测接口：

- src/predict_risk.py

功能：

- 输入体检指标 JSON（单条对象或对象数组）
- 自动按训练特征顺序补齐缺失字段
- 输出风险概率 risk_probability 和风险等级 risk_level
- 支持将结果保存为 JSON 文件

风险等级规则：

- low: 概率 < 0.33
- medium: 0.33 <= 概率 < 0.66
- high: 概率 >= 0.66

运行示例：

```powershell
python src/predict_risk.py --input-json data/input/user_case.json
```

指定模型和输出路径：

```powershell
python src/predict_risk.py --input-json data/input/user_case.json --model-path model/cardio_risk_multisource_ensemble.joblib --metadata-path model/cardio_risk_multisource_ensemble_meta.json --output-json data/output/prediction_result.json
```

输入 JSON 示例（单条）：

```json
{
	"age": 46,
	"sex": 1,
	"systolic_bp": 145,
	"diastolic_bp": 92,
	"bmi": 28.4,
	"total_cholesterol": 230,
	"fasting_glucose": 108,
	"smoker": 1,
	"alcohol": 0,
	"active": 0,
	"diabetes_history": 0,
	"hypertension_history": 1,
	"dataset_source": "user_input"
}
```
